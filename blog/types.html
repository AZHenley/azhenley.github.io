Type systems for the rest of us

8/2/2021

We have a variable with a value of 65.

What type is it?

We can probably agree it is an Integer.

It is obviously not a Bool or Char or String, or even a File or Person or Point3D.

But if it is an Integer then what about Number or Odd or i32?

Is it just one of them? Maybe it is all of them! (subtyping)

We'll just pretend it is something else when we need it to be. (type conversion)


Now, there will be a time when we need more values.

Not just one or two or three numbers, but k-numbers in some order.

When we pass this around, is it the same exact numbers or is it a new thing that just looks the same? (evaluation strategy)

Something else I would like to know, is this a List of Numbers or is this a List of k-Numbers? (dependent types)


Anyway, sometimes a list of numbers just won't work.

Can our variable hold either a value or nothing at all? (nullable types)

Better yet, I'll take either a List of Numbers or a Bool, but nothing else! (sum types)

We will just be mindful to know when it is which, and treat it as such.


Let's try to change the value.

Or must the value stay the same forever? (immutability)

Can we use the variable only once? Or twice? Is zero times ok? (substructural types)

Can it be in more than one place at once? (uniqueness types)


No matter! We can make our own types, right?

It shall consist of a u8 and a Bool, which gives us 256x2 different possibilities! (product types)

Types are better with names, so it shall be deemed Thing.

Should we allow just any ole u8 and Bool to be used as a Thing? (structural typing)

Or must it be specified as a Thing to be a Thing? (nominal typing)

**Types of types? Functions? Generics? Inheritance? Interface? Traits? Function pointers? Higher order functions? Function overloading? Effect types?

**Something.
